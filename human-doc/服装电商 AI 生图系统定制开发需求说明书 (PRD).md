# 服装电商 AI 生图系统定制开发需求说明书 (PRD)

## 一、 项目概述

### 1.1 项目背景
本系统面向鞋服全品类电商铺货业务，满足日上新数十至百款的高频制作需求。核心解决主图、场景图、详情细节图制作量大、人工美工成本高、出图慢、标准化差以及平台驳回率高（以唯品会为核心，覆盖淘宝、拼多多、京东、抖音）等核心痛点。

### 1.2 系统定位
打造企业内部专用的 “零提示词、全固化模板、傻瓜式一键出图” 的 AI 生图系统。
- 员工无需输入任何提示词与复杂的 AI 专业参数，全程可视化纯按钮/下拉框操作。
- 规则、风格、平台规范、服装标准全部由后台内置固化。
- 支持单款快速出图与大批量多图队列生成，输出图片接近精修标准，可直接上架。

### 1.3 技术方案演进与核心边界
本系统在技术实现上直接调用商用闭源 Image 2 图像大模型 API 作为核心生图引擎。
为了在调用云端通用大模型 API 的前提下，依然满足电商“高保真、货不对板控制、平台合规”的工业级要求，系统需在 API 基础之上封装一层“前置图像规整 + 提示词工程（Prompt Engineering）固化 + 后置像素级级联处理”的工程化中间件（Middleware），以弥补闭源大模型在像素级控图和私有化本地部署方面的原生局限性。

---

## 二、 系统架构与流程设计

系统整体采用前端操作轻量化、后端工程化包装 API 的级联架构。

### 2.1 核心业务流程
1. **员工上传服装原图**：提供平铺图或假人白底图。
2. **前置图像预处理**：系统利用分割模型自动提取服装 Mask，分离主体与背景。
3. **API 提示词组装**：后台提取固化的资产词库，拼接为符合 Image 2 API 的多模态 Prompts。
4. **调用 Image 2 API**：传入原图/Mask/高权重文本，执行高保真图生图或局部重绘。
5. **后置级联后处理**：执行高清修复(SR)、商用抠图融合（绝对值#FFFFFF白底）、全平台合规词/畸形二次检测。
6. **打包输出下载**：吐出主图套餐 / 详情页套餐，直接交付铺货上架。

## 三、 前端功能需求（员工操作端）

前端界面要求极简、电商人思维导向、无任何 AI 参数（如降噪强度、步数、种子等）。

### 3.1 素材上传区
- 支持单张服装原图（平铺图、白底图、衣架图、假人模特图）拖拽或点击上传。
- 支持批量上传（最高单次支持 100 款），系统自动进入批量多图队列。

### 3.2 平台与类目选择区
- **平台选择（单选/多选）**：唯品会、淘宝、拼多多、京东、抖音。
- **服装类目（单选）**：女装（连衣裙/外套/T恤/卫衣等）、男装、童装、鞋靴、箱包。

### 3.3 图片风格与场景选择区
- 采用全可视化卡片点选（包含缩略图示例）：
  - **外景街拍**：欧式建筑、韩式街道、都市高街、阳光户外。
  - **室内棚拍**：简约纯色、轻奢家居、现代工作室、自然光影窗边。
  - **高端画册**：极简艺术、杂志封面、复古胶片。

### 3.4 出图尺寸与数量设置
- **尺寸适配（多选）**：自动适配唯品会主图标准、淘宝 (800x800)、抖音竖版 (3:4/9:16)、详情页长图等。
- **出图数量套餐**：4张主图套餐、6张场景图套餐、9张详情全套套餐、自定义数量。

### 3.5 生成队列与结果下载区
- 提供异步任务进度条，展示当前排队位置与预估剩余时间。
- 结果支持单张预览、局部重绘微调（画布擦除重试）。
- 支持一键打包下载（ZIP 格式），下载包按“款式ID_平台_尺寸”自动命名归类。

## 四、 后台核心固化资产（系统核心价值）

所有资产库由系统管理员或运营团队在后台统一配置与维护，对前端操作员工完全屏蔽。

### 4.1 全平台合规规避词库
系统调用 API 时，在文本端及后置检测端自动加入负面约束，强制禁止生成：
- 极限词、违规营销文案、夸张乱入的第三方 Logo、畸形肢体/多指/异形、违规露骨姿势、杂乱背景、破坏平台审美的极端色调。

### 4.2 服装专属固定提示词工程 (Prompt Matrix)
根据前端员工选择的“服装类目+场景”，后端自动检索矩阵并拼接高权重文本注入 API：
- **面料真实性锁死**：纯棉/针织/牛仔/呢料/丝绸等面料的特定光泽度与纹理细化描述。
- **版型结构稳定**：宽松/修身/直筒/Oversize 的立体剪裁描述。
- **商业打光固化**：柔光、均匀三点打光、电商专业棚拍高端光影，确保输出质感。

## 五、 技术方案详解：基于 Image 2 API 的工业级落地适配

由于直接调用通用 Image 2 API 进行图生图容易产生衣服版型走样、纽扣/拉链幻觉等货不对板问题，后端必须采用以下三段式级联工程方案：

### 5.1 前置图像处理中间件 (Pre-processing)
- 系统接收到原图后，内置的轻量化分割网络（如 BiRefNet / SAM 2）自动运行，精准分离出服装的主体轮廓 Mask 遮罩。
- 识别服装的原生特征（如领型、袖子长度、颜色），作为元数据传递。

### 5.2 API 级联控图策略 (API Core Steering)
- **结构与版型保真**：在调用 Image 2 图像大模型的 image-to-image 或 inpainting 接口时，将原衣服图像作为强引导垫图。通过精确控制 API 暴露的强度系数（如将 strength 或重绘权重严格控制在大模型能识别但保留原图纹理的特定经验区间），将衣服区域进行高权重像素锁定。
- **局部遮罩重绘 (Inpainting)**：对于平铺图或带假人的原图，将服装主体区域生成 Mask 保护，仅将衣服以外的白底或衣架区域提交给 Image 2 API，命令其在 Mask 外部生成真人模特的肢体、面部及选定场景，实现衣服 100% 像素级保留，只换模特和背景。

### 5.3 后置图像级联后处理 (Post-processing)
- **纯白底首图标准化**：API 生成的白底图往往带有色偏。后处理模块自动对 API 的出图进行二次抠图，将背景强制填充为绝对纯白（#FFFFFF），并由算法自动在其脚底补齐符合唯品会/天猫机审规范的微浅灰色自然投影。
- **超分辨率增强 (SR)**：对 API 吐出的图像进行电商级的超分模型（如 Real-ESRGAN）二次放大与锐化，确保衣服面料材质的支数与纹理精细度达到美工精修级别。

## 六、 接口与算力成本管理（API 运营看板）

鉴于采用按量计费的云端 API 路线，系统后台必须内置调用成本管理模块：

### 6.1 成本与并发控制
- **调用限制**：支持针对不同的员工账号设置每日/每周最大出图额度，防止算力被无意义空转浪费。
- **异步队列缓冲**：前端请求全部写入 Redis 队列，后端根据 Image 2 API 的速率限制（Rate Limits）进行平滑并发请求，避免触发服务商的并发熔断。

### 6.2 账单与可用率看板
- 后台实时统计每日 API 消耗金额、出图成功率、平均生成耗时。
- 提供员工对出图可用性 (满意/不满意) 的反馈统计，不满意率超过阈值的风格模板自动提醒管理员优化提示词矩阵。
